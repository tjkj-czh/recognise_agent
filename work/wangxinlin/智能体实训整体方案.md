一、实训总目标

四层目标

第一层是业务目标。围绕遥感数据典型业务流程，验证大模型智能体在“任务理解、工具调度、空间分析、结果解释、异常提示、报告输出”上的真实价值，明确每个场景是否能替代部分人工流程、缩短多少作业时间、提升多少标准化程度。

第二层是产品目标。每位学员在5周内至少形成1个可独立演示的遥感场景智能体子产品，具备明确输入、稳定流程、结构化输出、可解释结果和可复用Skills，不是一次性脚本，而是具备产品雏形的Agent。

第三层是资产目标。沉淀一套遥感智能体能力资产，包括场景需求模板、技能模板、工具接口规范、评测集模板、复盘模板、平台适配模板，确保这次种子班结束后不是“人走项目散”，而是留下下一期可复用的方法论和组件库。

第四层是人才目标。让9位学员具备OPC式智能体产品落地能力，也就是能自己完成场景拆解、需求定义、工具接入、提示设计、评测构建、迭代优化、Demo包装和平台复现，而不是只会调用几个模型接口。

二、组织方式：3个产品小队 + 9个OPC岗位

每个小组固定为三个岗位。

第一个岗位是 Agent PM / 场景产品负责人。负责场景定义、需求边界、业务流程、输入输出规范、评测口径和最终Demo叙事。这个人要保证“做的是对的事”。

第二个岗位是 Agent Builder / 编排与集成负责人。负责智能体工作流设计、提示词工程、工具编排、Skills封装、异常处理、输出格式控制。这个人要保证“系统能稳定跑起来”。

第三个岗位是 Agent Evaluator / 评测与数据负责人。负责数据准备、样例构建、正负例设计、评测指标、迭代记录、问题归因。这个人要保证“效果是可测、可比、可优化的”。

这样每组三人刚好形成一个小型一人公司矩阵：一人盯市场和产品，一人盯研发和交付，一人盯运营和质量。虽然是三人协作，但每个人都能形成完整的OPC意识。

同时，在班级层面再设一个“横向中台机制”，推动三类共性能力。

一类是数据与工具中台，包括底图、影像、POI、DEM、公告文本、矢量格式、坐标系规范、统一接口规范。

一类是模型能力中台，包括分割、分类、变化检测、OCR、信息抽取、时序对齐等原子能力。

一类是评测中台，包括通用评测模板、用例标签体系、错误分类法、版本对照表。

这样可以避免三个组重复踩坑。

三、9位学员都要有独立好用的智能体的落地方式

采用“1个组级主Agent + 3个个人子Agent”的结构。

第一组辅助选址，可以拆成：

组级主Agent：井场辅助选址总控智能体

陈彬子Agent：空间约束核验Agent，专做缓冲区、禁建条件、红线碰撞检查

王垚飞子Agent：地形适宜性评分Agent，专做坡度、高程、通达性评分

曾可茹子Agent：选址解释与审查建议Agent，专做结论解释、人工复核点输出、报告生成

第二组用地识别，可以拆成：

组级主Agent：图斑用地识别总控智能体

张翔子Agent：图斑分割质检Agent，专做边界切分、分割质量诊断

程志慧子Agent：地类判别Agent，专做类别识别与置信度判断

王新林子Agent：合规审查提示Agent，专做基本农田、建设用地、水体等规则提醒

第三组变化监测，可以拆成：

组级主Agent：供地变化监测总控智能体

楼国华子Agent：公告抽取与地块定位Agent

徐文婕子Agent：时序变化比对Agent

邱深强子Agent：异常解释与预警报告Agent

这样最终不是3个成果，而是“3个主Agent + 9个个人Agent + 一批共享Skills”。从展示上也更完整：既能展示团队协同能力，也能展示个人独立能力。

四、成果定义

六件套交付

每个小组最终必须交付：

第一，1个组级主智能体。要求能完整跑通核心业务流程，有固定输入、可解释输出和标准演示脚本。

第二，3个个人子智能体。每个人必须能独立演示自己的Agent，不依赖组员在场解释。

第三，1套Skills资产包。至少分成三层：通用Skills、场景Skills、个人特色Skills。通用Skills如坐标系转换、GeoJSON合法性检查、遥感报告模板化输出；场景Skills如井场缓冲分析、地类矢量汇总、变化图斑筛查；个人特色Skills则对应个人子Agent的核心动作。

第四，1套评测集。不是简单测试样例，而是带标签体系的评测资产，至少包括标准样本、边界样本、异常样本、对抗样本、缺失信息样本。

第五，1套产品化文档。包括PRD-lite、工具清单、Skill说明、评测报告、版本迭代记录、已知问题说明。

第六，1套汇报与演示资产。包括5分钟演示脚本、业务价值页、技术架构页、效果页、风险页、下一步路线图。

五、三大场景设计

1\.辅助选址智能体：约束推理 + 多因子评分 + 审查协同的复合型Agent

输入定义为：地块范围、多类约束规则、POI与道路数据、地形数据、可选偏好参数。

中间链路分为五步：地块解析、约束检查、候选区筛选、适宜性评分、结论解释。这里强调“评分不是黑盒”，每个维度都要能追溯。比如民房距离、高速距离、学校距离、坡度等级、道路通达性分别占多少权重，要能解释“为什么这个点得分更高”。

输出必须至少有四类：候选位置清单、评分明细表、风险提示列表、人工审查建议。这样才适合真实业务使用。

这个场景的关键不只是空间分析，而是把“规则核验工具 + 空间计算工具 + 解释型输出”串成一个完整决策助手。

重点Skill包括：缓冲区分析Skill、叠置裁剪Skill、约束规则引擎Skill、适宜性评分Skill、选址报告生成Skill、人工复核清单生成Skill。

2\.用地识别智能体：影像理解 + 图斑判读 + 合规提示的判读型Agent

链路分成三层：视觉层、图斑层、规则层。

视觉层负责分割与分类。图斑层负责把像素/对象结果汇总到图斑层，形成面积、占比、主导类型、碎片化程度等统计。规则层负责把图斑判读结果映射到业务结论，例如“图斑中耕地占比超过X，且连续面积达到Y，需提示耕地保护关注”“低置信度区域位于边缘阴影区，建议人工复核”。

这样就从“模型输出”升级成“业务结论输出”。

重点Skill包括：图斑切片预处理Skill、分割结果清洗Skill、地类标签映射Skill、图斑属性汇总Skill、低置信区识别Skill、合规提示生成Skill。

3\.变化监测智能体：公告理解 + 时空对齐 + 异常归因的监测型Agent

定义三级能力

一级能力是公告结构化，把出让/划拨公告中的项目名称、位置、面积、时间、用途、边界描述提取出来。

二级能力是时空对齐，把公告时间点与影像时序、地块范围与影像覆盖关系对齐，判断是否具备检测条件。

三级能力是变化归因，不只是说“变了”，而是解释变的是哪一类变化，例如裸地转施工、农地转建设、水体消失、堆场新增等，并给出疑似风险标签。

重点Skill包括：公告文本抽取Skill、地块范围定位Skill、时序影像匹配Skill、变化图斑提取Skill、变化类型归因Skill、预警报告输出Skill。

六、建立遥感智能体能力分层图

在第一周建立一个遥感智能体的分层框架，后续所有设计都对齐这个框架。

第一层叫交互层。负责接收任务、追问缺失参数、输出报告、输出可视化结论。

第二层叫编排层。负责任务拆解、工具选择、步骤控制、异常回退、结果汇总，是智能体的“大脑”。

第三层叫能力层。负责调用具体的原子工具，比如空间缓冲、分割模型、变化检测、OCR、规则判断、评分计算。

第四层叫数据层。负责影像、DEM、POI、公告文本、矢量边界、规则库、评测样本。

第五层叫治理层。负责日志、评测、版本、规则口径、人工复核、平台复现。

这个分层非常重要，可以明白：Agent不是一个Prompt，而是一个“可编排、可验证、可治理”的产品系统。

七、Skills资产库结构

要把Skills当作“稳定的可复用工作单元”。一个好Skill应该具备明确输入、清晰执行逻辑、稳定输出格式、失败兜底策略和适用边界。

统一采用三级Skills库。

第一级是通用基础Skills。所有组都能复用，例如：

坐标系转换与检查

GeoJSON合法性检查

栅格/矢量格式标准化

空间范围裁剪

结果表格标准化输出

遥感任务参数补全

低质量输入告警

结果解释模板生成

第二级是遥感专业Skills，例如：

缓冲区约束分析

叠置分析与冲突检测

图斑切片与影像预处理

分割结果后处理

图斑统计汇总

时序变化比对

变化区域聚类

规则库驱动合规判定

第三级是场景专用Skills，例如：

井场适宜性评分

民房/学校/高速禁建核验

耕地保护识别提示

基本农田关注提示

公告地块解析

疑似未批先建预警

异常变化报告生成



每个Skill都统一模板，至少写清这几项：Skill名称、用途、适用前提、输入格式、处理步骤、输出格式、失败条件、人工接管条件、示例输入输出、评测要点。这样才真正能复用。

八、建立更专业的评测体系

1\.建立四层评测框架。

第一层是任务完成评测。看是否完成任务，是否输出了完整结果，是否漏掉关键步骤。

第二层是工具调用评测。看是否调用了正确工具、顺序是否合理、参数是否正确、失败后是否会重试或回退。

第三层是业务结果评测。看空间约束是否判断正确、分类结果是否符合标签、变化检测是否找到关键区域、报告结论是否合规。

第四层是产品体验评测。看输入要求是否清晰、解释是否能被业务人员理解、输出是否易于复核、处理时长是否可接受。

2\.每个场景都至少设置以下样本类型：

正常样本：标准流程可顺畅完成

边界样本：比如POI不完整、图斑跨边界、影像质量一般

异常样本：坐标错误、公告字段缺失、分类置信度极低

对抗样本：规则冲突、伪变化、阴影误判、地块描述模糊

鲁棒性样本：同一任务换一种说法、换一种输入顺序、换一个格式

3\.同时建议建立错误分类法，不然评测后很难迭代。比如错误分为：

任务理解错误

参数补全错误

工具选择错误

工具调用错误

空间逻辑错误

模型结果误用

规则映射错误

解释输出错误

异常兜底缺失

这样第三周以后每轮评测都能形成量化提升，不会停留在“感觉比上周好了”。

九、五周实训推进方案

强调“每周有目标、每天有动作、每组有产出、每人有责任”

第一周，主题是“定产品，不定代码”

目标不是急着开发，而是把场景边界、成功标准、输入输出和最小可行链路定义清楚。第一周做错，后面四周都会乱。

这一周每组要完成六件事。第一，完成场景PRD-lite，写清用户是谁、问题是什么、什么算完成。第二，定义主Agent与3个子Agent。第三，列出全部所需数据、模型、工具、规则。第四，梳理最小可行工作流，也就是不考虑最优，只考虑能跑通。第五，定义第一版评测口径。第六，完成平台和开发工具的熟悉。

助教这一周的关键动作不是讲技术，而是逼大家把问题定义具体。比如“用地识别”不能只说“识别耕地、水体、建设用地”，必须写清分类体系、最小上图面积、输出格式、谁来复核、什么场景不适用。

第一周末每组必须交付：一页场景卡、一张工作流图、一张数据工具表、一版评测框架图、一份演示任务样例。

第二周，主题是“定底座，不定炫技”

目标是把数据、工具和模型的可用性打通，形成稳定输入输出，不求智能体多聪明，先求原子能力可用。

这一周每组要完成四件事。第一，完成数据获取、清洗、格式统一、命名规范。第二，把模型组和微调组的能力封装成可调用原子工具，而不是口头能力。第三，做通用Skill封装。第四，建立第一版评测集。

这一周一定要强调“工具可调用性验证表”。每个工具都要记录：输入要求、输出格式、耗时、常见报错、失败场景、可替代方案。否则第三周集成时会崩。

第二周末每组必须交付：数据清单、工具接口说明、至少5个可调用Skills、至少15个评测样本、第一版错误台账。

第三周，主题是“做出能跑的Agent闭环”

目标是做出v0.1到v0.3，让主Agent和个人子Agent都能在小范围内跑通，并开始进入评测—修正—再评测闭环。

这一周建议采用日迭代节奏。周一做v0.1串联，周二评测，周三修正到v0.2，周四再评测，周五到v0.3并做阶段演示。

这一周要特别强调两个概念。第一个是“主链路优先”，不要一上来补所有边缘情况。第二个是“每次只优化一种问题”，否则会看不出版本进步来自哪里。

第三周末每组必须交付：主Agent v0.3、3个子Agent最小闭环、评测结果对比表、Top10问题清单、下一步优化策略。

第四周，主题是“做强效果，做实平台”

目标是把智能体从“能跑”提升到“可演示、可解释、可复现”，并开始在海纳平台中复现核心流程。

这一周重点不是继续加功能，而是围绕评测结果做“少而关键”的提升。比如辅助选址组要提高评分解释性；用地识别组要提高低置信区处理；变化监测组要提高公告抽取到影像对齐的稳定性。

同时这一周开始强制做平台复现，不能拖到第五周。因为很多项目最后死在“本地能跑，平台跑不顺”。

第四周末每组必须交付：优化后Agent、平台适配记录、第二版扩展评测集、用户演示脚本初稿、已知限制说明。

第五周，主题是“产品化封装与最终汇报”

目标是形成可演示、可讲清、可移交的最终成果。

这一周每组要完成：平台最终复现、演示脚本打磨、汇报PPT、成果文档归档、视频录制、个人复盘。

第五周我建议增加“个人答辩”机制。不是只按组汇报，而是每位学员要用5分钟回答三个问题：你的Agent解决了什么问题、你的Skill最核心的价值是什么、你的Agent当前最大的不确定性是什么。这样才能真正检验是否形成了OPC能力。

十、增加“每周例会机制”

每周一做目标会。确认本周唯一关键目标和交付物，不允许模糊。

每周三做中检会。每组用10分钟回答三个问题：现在跑通了什么、卡在哪里、需要什么支持。

每周五做评测与复盘会。不是展示“我们做了很多”，而是展示“本周哪3个指标变好了，哪2个问题还没解决”。

十一、给每位成员一个“OPC工作清单”

为了真正落实一人公司架构，每位成员必须在自己负责的子Agent上完成以下九项动作：

能讲清业务问题

能画出Agent流程

能说明调用了哪些工具

能写出至少2个Skill

能设计至少5个评测样本

能解释一次关键失败案例

能完成至少1轮效果优化

能做一次独立演示

能输出一页个人成果卡

这样到了结营时，不会有人只会说“我们组做了什么”，而能说“我做成了什么”。

十二、补充结营评价标准

公开评分标准

业务价值20分，看是否贴近真实遥感业务，是否能解释为什么值得做。

产品完成度20分，看输入输出是否完整，流程是否稳定，是否可演示。

Skill资产化20分，看是否形成复用能力，而不是一次性流程。

评测专业度20分，看是否有体系化评测、错误归因和版本迭代。

平台复现与表达20分，看是否能在海纳平台复现，是否能清楚讲明白。

这样大家会更重视“资产和评测”，而不只是“做个效果图”。

十三、预期成果

本次种子班最终不以单纯完成3个演示Demo为目标，而是以“3个组级主智能体、9个个人子智能体、1套遥感Agent Skills资产库、1套场景评测资产、1套平台复现方案、1套可复制的实训方法论”为核心成果框架。每个智能体均需具备明确场景边界、稳定工作流、可解释输出、评测记录和平台复现能力；每位学员均需形成独立可展示的子智能体与个人Skills成果，体现OPC式智能体产品落地能力。

十四、项目定位

本次实训并非传统模型调参或单点工具开发训练，而是面向遥感业务场景的“大模型智能体产品孵化实训”。其核心是以场景为牵引、以Skills为资产、以评测为闭环、以平台复现为交付，训练学员从0到1完成遥感智能体产品定义、能力组装、效果验证与应用落地的全流程能力。

十五、把“做项目”改成“打版本”

最容易犯的错误就是一直做、一直加，最后没有版本意识，采用版本里程碑：

v0.1：能跑通最小流程

v0.2：结果基本可用

v0.3：能解释、能演示

v0.4：评测稳定提升

v1.0：平台可复现、文档完整
