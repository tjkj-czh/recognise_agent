# 评估前置对齐

当用户想知道某个 skill 好不好用、有没有带来提升、或者多个同类 skill 谁更好时，先读这份文档。

默认把 `<skill-base>` 理解为当前 skill 的 `SKILL.md` 所在目录。

在这个项目里，`评估 / 测评 / 评测` 永远只代表一套标准化正式流程：前置对齐 -> 正式执行 -> `review.html` + `report.html`。前置对齐只是第一段，不是轻量版，也不是可以独立交付的替代路线。

- 一旦进入这份文档，先固定：`current_path` = `评估`。
- 执行过程中始终显式维护：`current_step` 和 `next_action`。
- 每次进入新步骤，或准备进入 benchmark、blind comparison、批量 eval 这类重型操作前，先复述：
  - 当前路径：`评估`
  - 当前步骤：`Step N`
  - 下一动作：一句话
- 如果任务已经从“评估输出质量”漂移成“优化触发描述”或“打包交付”，不要继续沿用这份文档；先回主 `SKILL.md` 的 Step 1 重新判路。
- 评估不是从打分开始，而是从 `判型 + 提案 + 对齐` 开始。
- 每次前置提案都必须同时给用户三样东西：
  - 我建议你怎么评
  - 除了这个，还可以怎么评
  - 这次你要我最终按哪种方式评

## 入口硬规则

- 只要用户第一次提出“评估 / 测评 / 评测 / 测一下某个 skill / 比较 skill 效果”这类请求，就必须先走这份文档。
- 只要用户说的是评估 / 测评 / 评测，就只能进入这套正式流程；不要改走结构判断、普通 review 或聊天式轻结论。
- 第一次响应默认只能停在 `AI 初判 + 评估前置提案 + 等用户拍板`，不要直接给分，不要直接说谁更好，也不要直接进入 `eval-loop.md`。这一步是在先对齐尺子，不是在把评估降级。
- `with-vs-without`、`single-skill`、`peer-vs-peer` 这三类请求都一样，先对齐尺子，再开始量长短。
- AI 自己写出来的推荐方案不算确认。只有用户明确回复“按这个评”“改成那个标准评”“主次方向就这样定”，才算真正锁定。
- 用户一旦明确拍板，就必须继续产出正式评估计划，并进入 `eval-loop.md` 执行到双 HTML 落地；不能把一段聊天结论当成完成。

## 什么时候必须先做前置对齐

满足下面任一条时，不要直接进 `eval-loop.md`：

- skill 的好坏存在明显主观空间，例如人格、语气、思维方式、方法论借用
- 用户说的是“像不像”“够不够像某个人”“是不是那个味儿”
- 你发现同一个 skill 至少有两种合理评法，而且结论可能完全不同
- 用户想比“有这个 skill 和没有这个 skill”的差别，但还没说清到底要看什么提升
- 用户想比两个或三个同类 skill，但还没锁定胜负标准

如果任务非常客观，例如只看通过率、耗时、成本，前置层也不能完全跳过；至少要写一个极简正式评估计划，把这次到底看什么写清楚。

## Step 1：先判型，不要先开跑

- 进入这一步时，更新：
  - `current_step` = `Step 1`
  - `next_action` = 先判断这个 skill 是什么类型，适合按什么方向评
- 如果这是用户第一次提出评估请求，先明确告诉用户：这一步还不是正式测评，而是在先对齐测评方案。
- 先读 skill 本体，判断它主要在增强什么：
  - 执行结果
  - 思考方式
  - 表达风格
  - 混合型价值
- 先给一个初步判型：
  - `落地效果型`：重点是活干得更好、更稳、更快、更省
  - `模仿还原型`：重点是像不像某种人物、语气、框架或行事方式
  - `混合型`：既要落地有效，也要保留某种思维 / 风格特征
- 如果它偏模仿，还要继续拆清楚到底在模仿什么：
  - 语气像不像
  - 表达习惯像不像
  - 思维框架像不像
  - 决策偏好像不像
  - 行事方式像不像
  - 结果风格像不像
- 因为 skill 默认是给 agent 看的，不是给人看的，所以只要目标不是明确的风格 / 人格模仿，默认优先考虑 `落地效果型`。
- 同时记录：
  - 这个 skill 最可能被什么错误标准误伤
  - 更适合做单 skill 体检、with-vs-without，还是同类 skill 横向比较
  - 哪些维度这次很可能不该混进主判定
- 默认把 `AI 初步判型` 固定成这 6 栏，不要自由发挥：
  1. `skill 是什么`：它主要在帮 agent 增强什么能力
  2. `主方向判断`：落地效果型 / 模仿还原型 / 混合型
  3. `次方向判断`：如果还有第二条价值线，就写出来
  4. `判断依据`：skill 里的哪些内容让你这样判断
  5. `容易误评的地方`：例如不该按语气像不像来打主分
  6. `建议下一步怎么对齐`：接下来最该问用户哪几个问题

## Step 2：给出评估前置提案，不只给一个建议

- 进入这一步时，更新：
  - `current_step` = `Step 2`
  - `next_action` = 基于初判写一份前置提案，把推荐评法和其他可选评法都摊给人看
- 这里不要只说“我建议这样评”，还要把“除了这个，还可以怎么评”明确亮出来。
- 这里讨论的是“这次用哪把尺子量”，不是在讨论要不要走轻量版；流程本身始终只有正式评估这一条。
- 如果你想直接照着固定骨架写，优先复制 `<skill-base>/assets/evaluation-proposal-template.md`。
- 如果你要的是“先怎么开口聊”的固定话术，优先复制 `<skill-base>/assets/evaluation-alignment-script.md`。
- 默认输出一份 `评估前置提案`，至少包含下面这些栏目：

```text
【评估前置提案】

1. 我初步判断这是什么 skill
- 它主要在解决什么问题
- 它更像哪一类：落地效果型 / 模仿还原型 / 混合型
- 它主要服务的是执行结果、思维方式，还是表达风格
- 我的判断依据是什么
- 我的判断把握有多大

2. 我建议优先按什么方向评
- 推荐主方向
- 推荐原因
- 这个方向最能看出它有没有价值

3. 除了这个，还可以按哪些方向评
- 可选方向 A：在看什么
- 可选方向 B：在看什么
- 可选方向 C：在看什么
- 哪些方向不要混着评
- 哪些方向这次不建议混在一起

4. 这次可能的主流评估方向
- 方向 1：落地效果
- 方向 2：模仿还原
- 如果是混合型，建议主次权重怎么分

5. 我建议的比较方式
- 单 skill 体检
- 有 skill vs 没 skill
- skill A vs skill B
- skill A vs skill B vs skill C
- 我推荐哪一种，为什么

6. 我建议重点看的维度
- 建议看的主维度
- 每个维度在看什么
- 每个维度为什么重要
- 哪些维度这次明确不看

7. 我建议怎么出测试题
- 什么类型的真实案例最能测出差别
- 为什么这种案例能测出差别
- 需要几类案例
- 哪些案例会制造假象，不建议用

8. 需要你来拍板的事
- 这次你最在意哪一种“好”
- 你明确不在意什么
- 你要的是单纯好用，还是要像某个人
- 如果要像某个人，你要像的是语气、思维、做事方式，还是结果风格
- 如果是混合型，你希望各自占多大比重
```

- 只要存在多种合理评法，就必须把“其他可能评法”也告诉用户，不要偷着替用户做完选择。
- 如果你已经很确定某个主方向，也要明确告诉用户“我为什么这样建议”，而不是只给结论。
- 这一步结束后，如果用户还没拍板，就停在这里等回复，不要自己继续往后写正式评估计划，更不要直接开始测评。这里是正式流程里的等待态，不是完成态。

## Step 3：和人类对齐，锁主方向和不看项

- 进入这一步时，更新：
  - `current_step` = `Step 3`
  - `next_action` = 让用户确认或修正评估目标，锁定主方向、次方向和不看项
- 这一轮要对齐清楚：
  - 主方向是什么
  - 是否有次方向
  - 如果有多个方向，各自权重是多少
  - 哪些维度明确不看
  - 比较方式是什么
  - 用什么类型的真实案例
- 如果用户说“我只看思维方式，不看语气”，要把“不看语气”也写成硬边界，不要后面又偷偷加回去。
- 如果用户说“落地效果最重要，但我也想保留一点某人的思考味道”，就把它记成 `主方向 + 次方向`，不要把两者混成一个大词。
- 对强主观任务，只要这一步还没对齐清楚，就不要继续进入正式评测。
- 对纯客观、低歧义任务，如果用户不在场，可以带着明确写出来的假设继续，但要把假设写进正式评估计划，不要默默脑补。
- 和用户对齐时，固定先说这 3 件事，再等用户拍板：
  - 我初步觉得它更适合按什么方式评
  - 除了这个之外，还可以按哪些方式评
  - 这次你到底想让我按哪一种来评
- 如果用户没有明确回答“按哪一种来评”，就视为这一步还没完成；不要把沉默、默认或 AI 自己的偏好当成确认。

## Step 4：产出正式评估计划

- 进入这一步时，更新：
  - `current_step` = `Step 4`
  - `next_action` = 把对齐结果压成一份正式评估计划，供后面的 eval loop 直接执行
- 只有用户已经明确拍板后，才允许进入这一步；不要把 AI 自己给出的推荐版直接写成“已确认计划”。
- 这份计划是给后面的执行层用的，不是给人聊天看的大段解释。
- 这份计划一旦锁定，下一步就只能进入正式执行；不要再把任务降回普通判断或聊天 review。
- 如果你想直接照着固定骨架写，优先复制 `<skill-base>/assets/evaluation-plan-template.md`。
- 默认输出一份 `正式评估计划`，至少包含：

```text
【正式评估计划】

1. 本次要评什么
- 评估对象是谁
- 是单 skill、with-vs-without，还是多 skill 对比

2. 本次最终采用的评估方向
- 主方向
- 次方向（如果有）
- 各自权重

3. 本次重点看的维度
- 维度 1
- 维度 2
- 维度 3
- 维度 4（可选）
- 维度 5（可选）

4. 本次明确不看什么
- 不纳入主判的项目
- 只作观察、不作主裁决的项目

5. 本次测试案例怎么选
- 真实案例来源
- 样本类型
- 样本数量
- 为什么这些样本能区分好坏

6. 本次怎么比较
- 同模型
- 同环境
- 同预算
- 同时间约束
- 是否盲评

7. 本次结论怎么写
- 总判断
- 分维度判断
- 证据来源
- 适用场景 / 不适用场景
- 哪些维度没提升
- 哪些维度反而退步
- 最后必须生成 `review.html` 和 `report.html` 两份 HTML
- `review.html` 保留基础证据工作台
- `report.html` 负责把计划、案例、回答、评分和结论讲给人看
```

- 主维度默认控制在 3-5 个；太多会把评估搞糊。
- 如果是 `混合型`，默认只保留 1 个主方向和 1 个次方向；不要一次摊三四条大主线。
- 如果需要机器写入格式，可以把正式评估计划保存成 `evals/eval-plan.json`；字段见 `<skill-base>/references/schemas.md`。
- 正式评估完成的硬标准，不只是“结果讲完了”，而是双 HTML 都已经落地。

## Step 5：决定现在是继续执行，还是先停在这里

- 进入这一步时，更新：
  - `current_step` = `Step 5`
  - `next_action` = 判断现在该进入执行评测，还是先停在前置层
- 如果用户当前还在讨论“到底该怎么评”，就停在这里，不要抢跑。
- 如果这次只是第一次收到评估请求，而用户还没明确确认方案，就默认停在这里，把前置提案发给用户，不要继续执行层。
- 如果正式评估计划已经锁定，再进入 `<skill-base>/references/eval-loop.md`；不要在这里直接给最终结论，也不要把 Markdown / 聊天结论当作正式评估完成。
- 如果真正的问题是“会不会触发”，而不是“触发后好不好”，回主 `SKILL.md` 的 Step 1，再决定是否走 `<skill-base>/references/description-optimization.md`。
- 进入 benchmark、blind comparison、批量 eval 前，再做一次状态播报，确认你是在执行已对齐的计划，不是在凭惯性加码。

## 主流评估方向速查

### 方向 1：落地效果

优先适用于：

- skill 本质上是在帮 agent 干活
- 用户最在意结果有没有提升
- 成本、速度、稳定性会影响最终价值

常用维度：

- 结果质量
- 任务完成率
- 正确性
- 稳定性
- 速度
- 成本
- 工具使用效率
- 性价比

### 方向 2：模仿还原

优先适用于：

- 用户明确想做某个人物 / 方法论 / 风格的 skill
- 好坏不只是看任务结果，还要看“像不像”

常用维度：

- 语气相似度
- 表达习惯相似度
- 思维框架相似度
- 决策偏好相似度
- 行事方式相似度
- 结果风格相似度

### 方向 3：混合型

适用于：

- 既要让 agent 干得更好
- 又要保留某种思考方式或风格特征

规则：

- 先定 1 个主方向，再定 1 个次方向
- 主次方向要写权重
- 主维度总数尽量别超过 5 个
- 如果用户没明确要求，不要默认把“语气像不像”混进“落地效果”

## 常见误区

- 不要让 AI 自己直接决定“什么叫好”，尤其是主观 skill。
- 不要把“语气像不像”和“落地效果好不好”默认混成一个分数。
- 不要在正式评估计划出来前，就急着写 assertions 或跑 benchmark。
- 不要用一套没对齐过的标准，去比较两个不同 skill 谁更好。
- 不要把断言通过率直接当成总胜负；主观维度和人类 review 仍然要回到前面的正式计划。
