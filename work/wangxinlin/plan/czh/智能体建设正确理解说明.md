# 用地识别智能体建设 — 正确理解说明

## 一、结论

你的理解**方向正确**，但还需要补充几个关键点。

你说的主线是：

- 先搭建系统框架
- 再制作 Skills
- 系统调用 Skills 完成任务
- 输出结果并回传系统整理后给用户展示

这条主线是成立的；在企业级落地中，还要加上“任务编排、数据契约、验证闭环、日志审计、可观测与降级”。

---

## 二、正确的完整理解（推荐口径）

用地识别智能体不是“只会调 Skill 的壳子”，而是一个可治理的业务系统：

1. **交互与任务入口层**：接收用户上传影像和任务参数。
2. **编排层（Agent Orchestrator）**：把任务拆解为步骤并调度执行顺序与异常回退。
3. **能力层（Skills/Services/Models）**：分割、分类、投票、VLM裁判、规则校验、统计等能力单元。
4. **数据与状态层**：保存过程数据、结果数据、任务状态、版本信息。
5. **治理层**：日志审计、监控告警、重试/回滚、权限与安全控制。
6. **展示与反馈层**：将结构化结果、统计报表、验证说明呈现给用户，并支持筛选与导出。

---

## 三、系统与 Skill 的关系（关键认识）

- **系统框架**负责“流程正确、状态可追踪、结果可治理”。
- **Skill**负责“单点能力稳定输出（有明确输入、输出、失败兜底）”。
- 系统不是被动调用 Skill，而是基于任务上下文和策略进行**有状态编排**。
- 最终交付给用户的是“经验证且可解释的业务结果”，不是单个 Skill 的原始输出。

---

## 四、推荐建设顺序（你可以按这个做）

### 阶段A：先搭系统底座

- 统一环境与依赖（Docker、数据库、对象存储、队列）
- 建立任务模型、状态机、日志链路（task_id/trace_id）
- 打通最小闭环：上传 -> 入库 -> 预览

### 阶段B：再做核心 Skills

- 图斑分割 Skill
- 多模型识别 Skill
- 投票验证 Skill
- 标签映射 Skill
- 统计聚合 Skill

### 阶段C：做验证增强 Skills

- VLM 裁判 Skill（仅处理争议图斑）
- 规则后校验 Skill
- 合规提示 Skill

### 阶段D：做结果产品化

- 前端可视化（地球渲染、状态样式）
- 报表导出（JSON/CSV）
- 可观测与告警（失败率、耗时、队列积压）

---

## 五、正确输出应包含什么

面向用户输出不应只包含“分类标签”，还应包含：

- 图斑级识别结果（主导类、占比、混合标记）
- 验证信息（投票结论、是否VLM裁判、规则校验结果）
- 统计汇总（按地类面积/占比/数量）
- 合规提示（参考性、非行政定性）
- 可追溯信息（任务ID、版本、时间）

---

## 六、常见误区（避免）

- 误区1：只做 Skill，不做编排与状态管理。
- 误区2：只输出模型结果，不做验证闭环。
- 误区3：只关注效果，不做日志审计和失败重试。
- 误区4：把合规提示写成行政定性结论（这是禁止的）。

---

## 七、一句话标准答案

你的理解总体正确；更准确地说，**要先搭建可治理的智能体系统底座，再将 Skills 作为能力插件纳入编排链路，通过“识别+验证+统计+提示”的闭环输出可信结果并回传前端展示**。
