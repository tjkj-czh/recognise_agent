## 一、目标与实施原则

### 1.1 目标

构建可在企业环境稳定运行的“用地识别智能体”系统，实现：

- 端到端自动化处理（上传→分割→识别→验证→统计→提示）
- 可解释与可追溯输出
- 安全、可运维、可扩展

### 1.2 原则

- **先可用，再高可用**：开发环境先跑通，生产环境再强化HA
- **先标准，再扩展**：统一数据契约和接口，再做模型扩展
- **配置优先**：模型、规则、阈值配置化
- **全链路治理**：日志、指标、告警、审计同步建设

## 二、统一技术栈（最终基线）

### 2.1 基础平台

- OS：`Ubuntu 22.04 LTS`
- GPU：`CUDA 12.4`
- Python：`3.10.x`（Conda 环境：`landuse`）
- Node.js：`20.x`，包管理：`pnpm`

### 2.2 应用层

- 前端：`Vue 3 + Vite + Element Plus + Cesium`
- 后端：`FastAPI + Uvicorn + Pydantic v2`
- 异步：`Celery + Redis`
- 空间处理：`GDAL + rasterio + geopandas + shapely`
- 推理：`PyTorch + transformers`
- VLM 服务：`vLLM + Qwen2-VL-7B-Instruct`

### 2.3 数据层

- 事务与空间：`PostgreSQL 16 + PostGIS 3.4`
- 对象存储：`MinIO`
- 缓存与队列：`Redis 7`

## 三、总体架构（企业落地）

```text
[Client(Web)]
   |
[API Gateway/Nginx]
   |
[FastAPI Service] --(sync)--> [PostgreSQL/PostGIS]
      |\
      | \--(async task enqueue)--> [Redis Broker] --> [Celery Workers]
      |                                     |--> 分割Worker
      |                                     |--> 分类Worker
      |                                     |--> 验证/统计Worker
      |
      +--(object io)--> [MinIO]
      +--(http)-------> [vLLM Service]

[Observability]
- Metrics: Prometheus
- Logs: JSON Logs / Loki(可选)
- Alerts: Alertmanager
```

### 3.1 部署分层

- **开发环境**：Docker Compose（单机）
- **测试环境**：Compose 或轻量 K8s
- **生产环境**：K8s（建议）+ Ingress + HPA + 持久化卷

### 3.2 组件边界

- `api-service`：鉴权、接口聚合、任务编排入口
- `worker-preprocess`：预处理、切片、元信息处理
- `worker-segment`：图斑分割与几何清洗
- `worker-classify`：多模型识别、投票、映射
- `worker-verify`：VLM裁判、规则校验、统计汇总
- `rule-engine`（可内嵌）: 合规规则匹配与提示生成

## 四、数据与接口设计

### 4.1 数据模型约束

- 图斑主键：`patch_id` 全链路唯一
- 必备追踪字段：`task_id, trace_id, created_at, updated_at`
- 版本字段：`model_version, rule_version, data_version`
- 几何字段：`geometry` 必建 `GIST` 索引

### 4.2 接口规范

- 路径风格：`/api/v1/...`
- 统一响应：`{"code":0,"message":"ok","data":{},"trace_id":"..."}`
- 错误码分层：参数/权限/业务/系统错误
- 上传接口：支持分片上传（生产建议）

### 4.3 幂等与重试

- 任务创建按 `idempotency_key` 去重（可选）
- Celery任务支持重试次数与指数退避
- 对外回调/推送应保证幂等

## 五、模型与验证策略

### 5.1 多模型策略

- 至少 2-3 种结构差异模型并行识别
- 输出完整投票上下文，禁止仅保留最终标签
- 模型切换通过配置中心或环境变量控制

### 5.2 VLM 裁判策略

- 仅对 `disputed` 图斑触发 VLM
- VLM请求设置超时与熔断
- 结果缓存（按图斑哈希 + 候选标签 + prompt版本）

### 5.3 规则后校验策略

- 空间、邻接、面积、标签冲突四类规则
- 规则版本化与灰度发布
- 规则失败不直接中断任务，标记“规则存疑”

## 六、安全与合规设计

### 6.1 凭据与密钥

- 禁止在仓库提交明文密码
- 开发环境使用 `.env`（仅本地），生产使用密钥管理服务
- 数据库/对象存储账户最小权限

### 6.2 网络与访问

- 生产启用 HTTPS
- 管理接口仅内网或受控网段访问
- MinIO、PostgreSQL、Redis 默认不暴露公网

### 6.3 审计与留痕

- 记录用户操作、任务状态变更、模型调用摘要
- 日志至少保留 180 天（按企业策略可调）

---

## 七、可观测与运维

### 7.1 监控指标

- API：QPS、延迟、错误率
- 任务：队列长度、执行耗时、失败率、重试次数
- 模型：推理耗时、GPU显存占用、超时率
- 数据：PostgreSQL慢查询、连接池占用

### 7.2 告警策略

- 任务失败率 > 5%（5分钟）
- 队列积压超阈值
- VLM服务连续不可达
- 数据库连接耗尽

### 7.3 备份恢复

- PostgreSQL：每日全量 + WAL 增量（或逻辑备份）
- MinIO：桶级版本管理与周期备份
- 恢复演练：至少每季度一次

## 八、交付计划（企业可落地）

### 阶段 0（第1周）：底座与规范

- 环境统一基线
- 数据库初始化与索引
- API骨架 + 上传入库 + 预览闭环

### 阶段 1（第2-3周）：核心识别链路

- 分割、识别、投票、映射
- 统计与基础报表
- Celery 编排与任务状态

### 阶段 2（第4周）：验证增强与可视化

- VLM裁判 + 规则后校验
- 前端验证状态可视化与筛选
- 关键路径性能优化

### 阶段 3（第5周）：企业化完善

- 监控告警、备份恢复、审计
- 安全加固、发布流程、回滚演练
- 版本封板与验收

## 九、发布与变更管理

### 9.1 CI/CD建议

- 代码检查：`black/isort/flake8`、`eslint/prettier`
- 测试门禁：单元测试 + 集成测试
- 镜像构建：按 `service:version` 打 tag
- 发布策略：蓝绿或金丝雀（至少支持快速回滚）

### 9.2 版本策略

- `vX.Y.Z`：主版本/特性/修复
- 规则、模型、数据版本独立编号
- 重大变更需附迁移说明

## 十、风险与应对

- **GPU资源竞争**：模型分时加载，任务并发上限控制
- **VLM抖动**：超时熔断 + 缓存 + 存疑降级
- **空间查询性能**：GIST索引 + bbox裁剪 + 批处理
- **数据不一致**：事务边界明确 + 幂等写入 + 对账任务

---

## 十一、最终决策清单（统一后）

- Python版本统一为 **3.10**
- 前端技术栈统一为 **Vue3 + Cesium**（不再使用“纯HTML/JS”口径）
- 生产默认 **不使用明文凭据**
- VLM仅用于争议图斑，默认具备降级策略
- 文档与实现以本方案为唯一基线
